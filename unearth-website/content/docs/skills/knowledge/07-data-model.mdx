## What This Skill Does

Discovers every data entity in the codebase -- what gets persisted, where it is stored, how it is shaped for different consumers, and which capabilities own it. Classifies entities by type (storage, projection, DTO, view model), maps storage mechanisms, traces cross-project projections, and identifies data model risks.

Without this step, the agent knows what the system does (capabilities) but not what data it manages. Data model understanding is essential for migration planning, database decomposition, and identifying hidden coupling through shared entities.

## Process Steps

1. **Load upstream context** -- Extract entity file locations, L1 capabilities for ownership mapping, and data store technologies from upstream outputs.
2. **Discover storage entities** -- Scan for persisted entities across all storage technologies (relational, document, graph, blob, cache) using ORM and framework patterns.
3. **Discover projections and DTOs** -- Catalog request DTOs, response DTOs, event payloads, view models, and projections that carry data between layers.
4. **Map entity attributes** -- Document the full attribute structure (fields, types, constraints, business meaning) for key entities shared across capabilities.
5. **Map entity relationships** -- Identify foreign keys, graph edges, embedded documents, implied references, and cross-store references.
6. **Map cross-project entity usage** -- Trace how entities flow across project boundaries (shared models, BFF projections, duplicate DTOs).
7. **Align entities to capabilities** -- Map each storage entity to its owning L1 capability and flag entities shared across 3+ capabilities as coupling hotspots.
8. **Identify data model risks** -- Review for missing primary keys, implied references without constraints, cross-store references, schema-less risks, and high-coupling entities.
9. **Compile data model output** -- Assemble all findings into the final output file.

## Key Output Sections

| Section | Content |
|---------|---------|
| Entity Inventory | All storage entities and projections/DTOs with file paths, storage technology, and type classification |
| Entity Attribute Details | Field-level documentation for key entities with constraints and business meaning |
| Entity Relationships | Relationship map including explicit FKs, graph edges, and implied references |
| Cross-Project Entity Usage | How entities appear across core API, BFF, and frontend in different forms |
| Capability Alignment | Entity-to-capability ownership mapping with coupling hotspot identification |
| Data Model Findings | Missing constraints, cross-store references, and schema risks with severity |

## Verification Gate

Before marking complete, the output must pass:

- All storage technologies from upstream are represented in the entity inventory
- Storage entities are discovered with file paths, storage type, and collection/table names
- Projections, DTOs, and view models are cataloged separately from storage entities
- Entity relationships are mapped (explicit FK/edge and implied references)
- Every storage entity is aligned to an owning L1 capability
- Coupling hotspots (entities shared by 3+ capabilities) are flagged with risk assessment
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
