## What This Skill Does

Maps the full dependency landscape -- how components depend on each other internally, what external packages they consume, and where shared libraries create coupling. Identifies version risks, EOL frameworks, dead dependencies, and critical coupling points.

Without this step, the agent understands patterns and structure but not the specific dependency chains that constrain evolution. A god class cannot be decomposed if 6 other components depend on it. A framework cannot be upgraded if conflicting versions exist across containers. Dependencies determine what is practically possible in modernization.

## Process Steps

1. **Load upstream context** -- Extract component inventories, package-to-system mappings, and shared library lists from upstream outputs.
2. **Map business-to-business dependencies** -- Trace dependencies between business components within each container (direct calls, shared repositories, shared data stores, event coupling, HTTP calls) and flag circular dependencies.
3. **Map business-to-technical dependencies** -- Identify which technical components (auth, ORM, validation, resilience, logging) each business component depends on, and compute fan-in for each technical component.
4. **Map external package dependencies** -- Inventory all external packages across every project with version, purpose, and risk flags (EOL, major version behind, pre-release, version conflict, unused).
5. **Analyze shared library coupling** -- For each shared library, assess what it provides, which containers consume it, coupling severity, interface stability, and extraction difficulty.
6. **Identify critical coupling points** -- Synthesize findings to flag components with fan-in greater than 5, circular dependencies, shared databases, business logic in shared libraries, version conflicts, and EOL frameworks.
7. **Produce dependency statistics** -- Compile summary metrics for all dependency categories.
8. **Compile dependencies output** -- Assemble all findings into the final output file.

## Key Output Sections

| Section | Content |
|---------|---------|

## Verification Gate

Before marking complete, the output must pass:

- Business-to-business dependencies are mapped for all analyzed containers
- External packages are inventoried for every project with version and risk flags
- Version conflicts across projects are identified
- EOL or deprecated frameworks are flagged with dates and affected containers
- Shared libraries are analyzed for coupling severity and interface stability
- Critical coupling points are identified with impact and constraint assessment
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
