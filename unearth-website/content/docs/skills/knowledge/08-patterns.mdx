## What This Skill Does

Detects the architecture patterns and anti-patterns present in the codebase by synthesizing evidence from structural analysis (containers, components) and knowledge extraction (capabilities, data model). Each pattern is classified with evidence, scope, and an assessment of whether it is well-implemented, partial, or problematic.

Without this step, the agent can describe what the system does and how it is structured, but cannot characterize how well it is structured. Pattern detection reveals intentional design decisions, accidental complexity, and structural risks that directly inform tech debt assessment and modernization planning.

## Process Steps

1. **Load upstream context** -- Extract container topology, component layer structures, and data model findings from upstream outputs.
2. **Detect structural patterns** -- Scan for architecture structure patterns (Layered N-Tier, Clean Architecture, CQRS, Microservices, BFF, Event-Driven, Serverless, and others) with specific evidence signals.
3. **Detect domain model patterns** -- Identify how the domain is modeled (anemic model, rich model, DDD, active record, data mapper, entity-per-table).
4. **Detect anti-patterns** -- Catalog structural problems including god classes, fat controllers, circular dependencies, shared databases, distributed monoliths, and missing abstractions with severity ratings.
5. **Detect communication patterns** -- Analyze how containers communicate (synchronous chains, async decoupling, sagas, orchestration, circuit breakers, resilience gaps).
6. **Detect security patterns** -- Characterize the authentication and authorization architecture (centralized auth, gateway-level auth, RBAC, per-service auth).
7. **Compile pattern assessment** -- Produce a structured assessment for each pattern with name, category, scope, evidence, assessment, and impact.
8. **Compile patterns output** -- Assemble all findings into the final output file.

## Key Output Sections

| Section | Content |
|---------|---------|
| Pattern Summary | Master table of all detected patterns and anti-patterns with category and assessment |
| Structural Patterns | Detailed assessment of each architecture pattern with evidence from multiple upstream outputs |
| Domain Model Patterns | How entities and business logic are organized (anemic vs. rich, DDD indicators) |
| Anti-Patterns | Cataloged structural problems with severity, evidence, impact, and remediation guidance |
| Communication Patterns | Inter-container communication style (sync chains, async, resilience) |
| Security Patterns | Auth architecture characterization (centralized, gateway, per-service) |
| Absent Patterns | Patterns expected for this type of system but not found, with impact assessment |
| Pattern Interaction Analysis | How detected patterns complement, conflict with, or amplify each other |

## Verification Gate

Before marking complete, the output must pass:

- Structural patterns are detected with evidence from both container and component analysis
- Domain model patterns are detected with evidence from data model and component analysis
- Anti-patterns are cataloged with file-level evidence and severity ratings
- Communication and security patterns are detected from upstream topology and auth analysis
- Each pattern has name, category, scope, evidence, assessment, and impact
- Absent-but-expected patterns are documented with impact assessment
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
