## What This Skill Does

Compiles a single, prioritized tech debt register by harvesting findings from every upstream skill output. Rather than discovering new issues, this skill synthesizes and classifies the god classes, test gaps, EOL frameworks, missing resilience, coupling hotspots, and anti-patterns already identified across the pipeline into a structured register with severity, effort, and remediation guidance.

Without this step, tech debt findings are scattered across 10 upstream outputs. Stakeholders see individual problems but cannot prioritize across them. This skill consolidates everything into one register that answers: "What should we fix first, and why?"

## Process Steps

1. **Harvest Findings from Upstream Outputs.** Systematically scan each upstream output for findings that represent tech debt, recording the source output and section number for traceability.
2. **Deduplicate and Consolidate.** Merge findings that describe the same underlying issue across multiple outputs, keeping the most specific evidence.
3. **Classify Each Debt Item.** Assign category (code quality, test coverage, architecture, resilience, security, dependency, data model, redundancy), severity (Critical/High/Medium/Low), effort estimate (Small/Medium/Large/Extra-Large), and blast radius.
4. **Assign Debt IDs and Group.** Give each item a unique ID using the format `TD-{category}-{number}` (e.g., `TD-CQ-01` for code quality, `TD-SE-02` for security).
5. **Calculate Debt Summary Statistics.** Compile totals by severity, category, and effort. Identify the top 5 most affected components.
6. **Build Remediation Roadmap.** Organize items into Quick Wins (high impact, low effort), Planned Work (high impact, medium effort), Strategic Items (high impact, high effort), and Deferred (low impact or blocked).
7. **Compile Tech Debt Output.** Assemble all findings into the output file.

## Key Output Sections

| Section | Content |
|---------|---------|
| Debt Register | All debt items grouped by severity, each with ID, category, source, affected components, effort, and evidence |
| Summary Statistics | Totals by severity, by category, by effort, estimated dev-weeks, and most affected components |
| Remediation Roadmap | Quick wins, planned work, strategic items, and deferred items with prerequisites and expected outcomes |

## Verification Gate

Before marking complete, the output must pass:

- Findings are harvested from at least 4 upstream outputs
- Items are deduplicated (no finding appears twice in the register)
- Each item has: ID, description, category, source, affected components, effort, and evidence
- Severity levels are differentiated (not all HIGH)
- Summary statistics reconcile with the detailed register
- Remediation roadmap covers all items (quick wins + planned + strategic + deferred = total)
- Each roadmap item has prerequisites identified (or explicitly "none")
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
