## What This Skill Does

Maps every source file through a traceability chain: file to layer to business component to L1 capability to container. Creates a matrix that answers "which files belong to which capability?" and "if I change this file, what is affected?" Also produces migration guides per component by analyzing extraction difficulty.

Without this step, the mapping from code to capabilities exists only in the agent's intermediate reasoning. Making it explicit enables impact analysis, migration planning, and team-level ownership assignment.

## Process Steps

1. **Load Upstream Context.** Read component file-by-layer tables from S-05, L1/L2 capability definitions from S-06, and container inventory from S-04. Build the traceability chain: File to Layer to Component to Capability to Container.
2. **Build the Traceability Matrix.** For each analyzed container, map every source file to its layer, business component, technical component, L1/L2 capability, container, test status, and any associated debt items.
3. **Validate Completeness.** Compare file counts against S-00 recon and S-05 analysis. Identify orphan files not assigned to any component and resolve them.
4. **Generate Per-Component Traceability Tables.** For each business component, produce a focused table showing all files grouped by layer with test coverage and cross-component dependencies.
5. **Assess Migration Readiness per Component.** Rate each component as Easy, Medium, or Hard to extract, documenting the key blocker, dependencies to resolve, files that move, and files that stay.
6. **Build Impact Analysis Indexes.** Create a Capability Impact Index (which files implement each L1) and a File Impact Index (high-coupling files with outsized blast radius).
7. **Compile Traceability Output.** Assemble all findings into the output file.

## Key Output Sections

| Section | Content |
|---------|---------|
| Traceability Chain | Summary metrics: total files traced, component counts, capability counts, orphan count |
| Per-Component Traceability | Files grouped by layer for each business component with test coverage and cross-component dependencies |
| Technical Component Traceability | Cross-cutting infrastructure mapped to the capabilities that use it |
| Migration Readiness | Extraction difficulty per component with key blockers and dependency details |
| Capability Impact Index | For each L1 capability, all containers, components, and file counts that implement it |
| File Impact Index | High-coupling files (god classes, shared repositories, core entities) with their blast radius |

## Verification Gate

Before marking complete, the output must pass:

- Every production source file from S-05 appears in the traceability matrix
- Each file is mapped to: layer, component (BC/TC), capability (L1/L2), and container (CON)
- Orphan files are identified and resolved (assigned or classified)
- Migration readiness is assessed for each component (Easy/Medium/Hard with key blocker)
- Capability impact index maps every L1 to its containers, components, and file count
- File impact index identifies high-coupling files with their blast radius
- Cross-component dependencies are noted in per-component tables
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
