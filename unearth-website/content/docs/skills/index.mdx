## Skill Anatomy

Every skill follows the same 10-section structure. This consistency means once you understand one skill, you understand them all.

1. **YAML Frontmatter** -- name and description for tool integration
2. **Pipeline Position** -- stage, order, upstream/downstream dependencies
3. **Inputs** -- required and optional data from previous skills
4. **Process Steps** -- numbered steps the agent follows literally
5. **Checkpoints** -- mandatory verification pauses between steps
6. **Output Format** -- exact Markdown structure the output must follow
7. **Evidence Standards** -- what counts as proof for each claim type
8. **Common Rationalizations** -- shortcuts to resist (and why)
9. **Red Flags** -- signs the output is incomplete or wrong
10. **Verification Gate** -- checklist the output must pass before the next skill runs

See the [full Skill Anatomy reference](/docs/reference/skill-anatomy) for details on each section.

## Chaining: How Skills Connect

Each skill declares exactly which upstream outputs it reads and which sections it needs:

- **Skill 00 (Recon)** reads the file system directly -- no upstream inputs
- **Skill 01 (Entry Points)** reads Section 4 of `outputs/00-recon.md` (Project Organization)
- **Skill 06 (Capabilities)** reads Section 3 from every `outputs/05-components-*.md` file (Business Components)
- **Skill 13 (Documentation)** reads all 12 upstream outputs to synthesize the final document

This explicit chaining means the pipeline is reproducible -- the same inputs produce the same outputs, and you can re-run any skill to refresh its output without affecting others.
