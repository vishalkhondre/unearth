## What This Skill Does

Identifies every human and system actor that interacts with the codebase, then draws the line between what is inside the system and what is outside. This produces the actor inventory, the system boundary definition, and the actor-to-system relationship map -- the three inputs required for a C4 Context diagram.

Without this step, the agent cannot distinguish internal components from external dependencies, and will conflate users, partners, internal services, and monitoring probes into a single undifferentiated "caller."

## Process Steps

1. **Load upstream context** -- Read auth pattern summaries and entry point tables from `01-entry-points` and the project list from `00-recon`
2. **Identify human actors from auth patterns** -- Group entry points by auth mechanism (JWT with RBAC, session/cookie, OAuth consent) to discover distinct user roles with evidence
3. **Identify system actors from auth patterns** -- Find machine-to-machine callers via API keys, client credentials, managed identity, and subscription key patterns
4. **Identify implicit actors** -- Detect actors not visible from auth alone: event producers, scheduled triggers, blob uploaders, change feed sources, and notification recipients
5. **Draw the system boundary** -- Classify every component as inside or outside the boundary using source code ownership, deployment pipeline, and managed service criteria
6. **Map actor-to-system relationships** -- Record each actor's entry point, protocol, direction, and description of what they do
7. **Identify open questions** -- Flag boundary ambiguities, actor distinctions, and ownership questions that need architect or product owner input
8. **Compile actors and boundaries output** -- Assemble findings into the output file

## Key Output Sections

| Section | Content |
|---------|---------|
| Human Actors | Each actor with name, description, auth pattern, entry points, evidence, and confidence level |
| System Actors | Machine-to-machine callers with the same attributes as human actors |
| System Boundary | Tables of what is inside vs. outside the boundary, plus ambiguities with resolution questions |
| Actor-to-System Relationships | Every actor mapped to entry points with protocol and direction |

## Verification Gate

Before marking complete, the output must pass:

- Every auth pattern from `outputs/01-entry-points.md` Section 8 is mapped to an actor
- Every human actor has name, description, auth pattern with evidence, entry points, and confidence level
- Every system actor has the same attributes
- Every project from `outputs/00-recon.md` is classified as inside or outside the boundary with justification
- Boundary ambiguities are flagged with specific resolution questions
- Actor-to-system relationships include protocol and direction for every actor
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
