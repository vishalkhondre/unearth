## What This Skill Does

Scans the repository structure to build a grounded mental model of the codebase before making any architectural claims. Detects programming languages, frameworks, build systems, project organization, and file distribution. For each technology detected, it records the specific config file that proves the claim and the version where detectable.

Without this step, the agent risks misidentifying the tech stack, missing entire projects in a monorepo, or making assumptions that compound errors through every subsequent skill.

## Process Steps

1. **Enumerate repositories in scope** -- List every repo included in the discovery with its name, path, and remote URL
2. **Scan root structure** -- Classify every top-level file and directory as source code, configuration, build system, CI/CD, containerization, infrastructure, documentation, test, tooling, or dependency lock file
3. **Detect technology stack** -- Identify languages, frameworks, and runtime versions from build and config files, citing the specific file path as evidence for each claim
4. **Identify project organization** -- Determine whether the codebase is a single project, multi-project, monorepo, or multi-repo, and list every sub-project with its technology and role
5. **Count and classify files** -- Count files per project by category (source, test, config, markup, docs, build) and flag notable ratios such as low test-to-source coverage
6. **Detect build system and CI/CD** -- Identify the build tool, CI/CD pipelines, containerization status, and infrastructure-as-code presence with file path evidence for each
7. **Scan for anomalies and surprises** -- Flag mixed framework versions, dead projects, README contradictions, ghost dependencies, and anything unexpected
8. **Compile recon output** -- Assemble all findings into the output file following the required structure

## Verification Gate

Before marking complete, the output must pass:

- Every repository in scope is listed with its path
- Every detected language has a file extension count AND a config file citation
- Every detected framework has a config file path AND a version where available
- Project organization type is determined with structural evidence
- File counts are recorded per project for source, test, config, and docs
- Build system, CI/CD pipeline, and containerization status are each determined (present with details, or explicitly absent)
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
