## What This Skill Does

Identifies every independently deployable runtime unit within the system boundary -- the C4 "container" level. Maps how containers communicate with each other and with external systems. Detects architecture patterns that emerge from the container topology, such as Backend-for-Frontend, shared database, event-driven, and synchronous chain patterns.

Without this step, the agent cannot distinguish deployable units from code modules, will miss inter-service communication patterns, and will lack the structural foundation needed for component-level analysis in `05-components`.

## Process Steps

1. **Load upstream context** -- Read the project list, entry point counts, boundary definition, and external system inventory from all four upstream outputs
2. **Identify application containers** -- Classify each inside-boundary project as a container (has its own deployment target, entry points, or process startup) or non-container (shared library, test project, build tool)
3. **Map shared libraries and cross-cutting code** -- Identify code shared across containers with which containers reference it and what it provides
4. **Map container-to-container communication** -- Find HTTP client configs, message producers/consumers, shared database access, and other inter-container links with protocol and direction
5. **Map container-to-external-system communication** -- Connect each container to its external systems from `03-external-systems` with protocol, direction, and resilience status
6. **Detect architecture patterns** -- Identify BFF, API gateway, shared database, event-driven/CQRS, synchronous chain, microservices, monolith, and strangler fig patterns with evidence
7. **Identify key findings** -- Flag synchronous chains longer than 3 hops, shared databases, missing resilience, single points of failure, and mixed framework versions
8. **Compile containers output** -- Assemble findings including an ASCII topology diagram and 3-5 key request flows

## Verification Gate

Before marking complete, the output must pass:

- Every inside-boundary project from `02-actors-and-boundaries` is classified as container or non-container
- Every container has ID, name, technology, type, entry point count, responsibility, and deploy target
- Shared libraries are identified with their consumers listed
- Every container-to-container communication link is mapped with protocol, direction, and evidence
- Architecture patterns are detected and assessed
- Key request flows (3-5) are documented end-to-end
- ASCII topology diagram is included
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
