## What This Skill Does

Deep-dives into a single container to map its internal structure at the C4 component level. Identifies business components (domain logic units), technical components (cross-cutting infrastructure), internal layers, god classes, test coverage, and business rules. This skill is repeatable -- run it once per container selected for analysis.

Without this step, the agent sees containers as black boxes -- it can describe what they do at a high level but cannot identify internal coupling, god classes, untested code paths, or business rule locations that are critical for capability extraction and tech debt assessment.

## Process Steps

1. **Select and scope the container** -- Choose a container from `04-containers` and record its ID, source directory, technology, entry point count, and external connections
2. **Map internal layer structure** -- Identify architectural layers (entry/presentation, business/application, domain, data access, models/DTOs, validation, configuration, infrastructure, cross-cutting) with directory paths and file counts
3. **Identify business components** -- Group files by business function by tracing from entry points through services, repositories, and models to find cohesive units
4. **Identify technical components** -- Catalog cross-cutting infrastructure (authentication, authorization, data access, validation, error handling, logging, configuration, HTTP clients) with which business components use them
5. **Catalog business rules** -- Extract concrete validation rules, value constraints, state transitions, calculations, authorization rules, and domain invariants with the class and method where each is enforced
6. **Detect god classes and complexity hotspots** -- Find files over 500 lines, classes with more than 10 public methods, deep nesting, and high coupling, with exact line counts and severity
7. **Assess test coverage** -- Count test files and methods, identify the test framework, calculate test-to-source ratio, and note coverage gaps per business component
8. **Map intra-container dependencies** -- Record business-to-technical, business-to-business, and business-to-external dependencies with coupling mechanisms
9. **Compile components output** -- Assemble findings into the per-container output file

## Verification Gate

Before marking complete, the output must pass:

- Container is correctly scoped (right directories, right files)
- Every source file is assigned to a layer
- Business components are identified by business function, not technology
- Each business component has responsibilities, files by layer, business rules, technology stack, and known issues
- God classes are detected with exact line counts and severity
- Test coverage is assessed per component with specific gaps identified
- Intra-container dependencies are mapped (business-to-technical, business-to-business, business-to-external)
- All uncertain findings are flagged as `[NEEDS VERIFICATION]`
