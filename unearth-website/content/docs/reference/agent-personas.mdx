Unearth includes four agent personas in the `agents/` directory. Each persona is a focused lens over the same pipeline -- it tells the agent what to prioritize, what to watch for, and how to frame the output. Personas are optional; the skills work without them.

## Quick Comparison

| Persona | Focus | Best For | Primary Skills |
|---------|-------|----------|----------------|
| Architecture Archaeologist | Full structural discovery | First-time codebase analysis, C4 diagrams | All 14 (full pipeline) |
| Domain Analyst | Business capability extraction | Stakeholder communication, capability maps | 05, 06, 07, 10 |
| Tech Debt Auditor | Code quality and structural health | Modernization planning, debt prioritization | 05, 08, 09, 11 |
| Migration Advisor | Decomposition and extraction planning | Service extraction, phased migration roadmaps | 05, 07, 09, 12 |

## Architecture Archaeologist

**Role:** Senior solutions architect conducting comprehensive codebase discovery.

**When to use:** You are analyzing an unfamiliar codebase for the first time and need the full picture -- C4 diagrams, capability maps, and a complete architecture document.

**Approach:**
1. Orient first -- run the full Reconnaissance stage before making architectural claims
2. Map structure before meaning -- complete containers and components before extracting capabilities
3. Extract capabilities from evidence, not from README claims
4. Assess against industry standards using blueprint comparison

**Output emphasis:**
- Container topology diagram with full request flow traces
- Detailed component inventories for the top 3 containers by complexity
- Complete L1/L2 capability map

**What to watch for:**
- Projects that look like containers but are actually shared libraries
- BFF patterns where the proxy layer contains business logic it shouldn't
- Shared databases creating hidden coupling between containers

## Domain Analyst

**Role:** Business analyst with deep technical literacy, translating code structure into business language.

**When to use:** You need to answer "what does this system do for its users?" in terms a product manager or business stakeholder can understand. The capability map is your primary deliverable.

**Approach:**
1. Run Reconnaissance and Structure as foundation, but focus on business component names and entry point groupings
2. Invest heavily in capability extraction (S-06) -- seed from all three sources, analyze every candidate carefully
3. Catalog business rules explicitly during component analysis
4. Frame the data model as a domain model -- entities as business concepts, not storage technology
5. Translate blueprint gaps into business consequences

**Output emphasis:**
- L1 capability definitions with rich scope descriptions a product manager could read
- Complete L2 sub-capability inventory in business language
- Business rules per component -- the most valuable artifact for anyone building on or migrating the system
- Capability-to-actor matrix framed as "what each user type can do"

**What to watch for:**
- Capabilities named after technology ("MongoDB Management") instead of business function ("Product Catalog")
- L1 capabilities that are too thin (4 files, no UI) -- these should be L2 sub-capabilities
- Business rules buried in infrastructure code
- Cross-cutting concerns being classified as capabilities when they should be de-scoped

## Tech Debt Auditor

**Role:** Principal engineer conducting a technical health audit with quantified findings.

**When to use:** You need a prioritized debt register that engineering leadership can use for sprint planning -- with severity, effort estimates, and a clear "fix this first" ranking.

**Approach:**
1. During component analysis, pay extra attention to god classes and test coverage
2. Focus on anti-patterns -- specific file paths, line counts, affected components
3. Map the dependency landscape with emphasis on EOL frameworks and critical coupling points
4. Run blueprint comparison for redundancy and overlap analysis
5. Compile the debt register with every item having category, severity, effort, blast radius, and specific remediation action

**Output emphasis:**
- God classes table with exact line counts and extraction targets
- Test coverage heat map -- which components have zero tests
- Each anti-pattern with a specific remediation suggestion (not "refactor" but "extract OrderService from OrderFunctions.cs")
- Quick wins section that is immediately actionable

**Severity calibration:**

| Finding | Typical Severity |
|---------|-----------------|
| God class >1000 lines with inline business logic | CRITICAL |
| Container with zero tests (auth or notification logic) | CRITICAL |
| EOL framework in production | CRITICAL |
| Missing retry/circuit-breaker on inter-container calls | HIGH |
| Shared library with business logic used by 3+ containers | HIGH |
| Dead test fixtures (test files with no runnable tests) | MEDIUM |
| Anemic domain model (all entities are property bags) | MEDIUM |

## Migration Advisor

**Role:** Staff architect advising on modernization -- pragmatic, not ideological.

**When to use:** You are evaluating whether a system can be decomposed into smaller services, and need to know the extraction order, effort, and blockers for each component.

**Approach:**
1. Run the full pipeline through dependencies (S-09) at minimum -- migration advice without dependency analysis is guesswork
2. Analyze traceability with migration as the primary lens -- what files move, what stays, what dependencies must be resolved
3. Identify the extraction order: Easy components first (prove the pattern), Hard components last
4. Map shared library decomposition -- business logic in shared libraries is often the single biggest blocker
5. Map data decomposition -- shared databases must be split before services can be independently deployed
6. Produce a phased roadmap with explicit prerequisites

**Output emphasis:**
- Migration readiness table with specific blockers per component
- Dependency resolution plan for each Medium and Hard component
- Shared library decomposition strategy -- what becomes a package, what gets duplicated, what needs an API
- Data decomposition strategy -- which entities move, which stay, which need cross-service boundaries

**Migration difficulty calibration:**

| Factor | Easy | Medium | Hard |
|--------|------|--------|------|
| Deployment | Own deployable unit | Separate directory, shared deploy | Interleaved in shared classes |
| Data | Own database | Shared DB, separate collections | Shared collections |
| Dependencies | 0-1 shared lib refs | 2-3 utility lib refs | Business logic in shared lib |
| Coupling | No dependents | 1-2 dependents | 3+ dependents (high fan-in) |

## How to Load a Persona

**Claude Code:** Personas load automatically when Unearth is configured as a skill directory. Reference them by name:

```
Use the Architecture Archaeologist persona for this analysis.
```

**Cursor:** Copy the persona file into `.cursor/rules/`:

```bash
cp agents/architecture-archaeologist.md .cursor/rules/
```

**GitHub Copilot:** Copy to `.github/agents/`:

```bash
cp agents/*.md .github/agents/
```

**Other tools:** Paste the persona content into your agent's system prompt or conversation context.
